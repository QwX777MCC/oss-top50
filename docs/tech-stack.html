Learning Notes · 技术栈梳理

oss-top50 技术栈全览

从零开始理解一个完整的前端项目：数据获取 → 脚本构建 → 静态页面 → 自动部署。每个层次配新手解读，边看边学。

一 项目全景图

一句话定义

一个纯静态的 GitHub 开源项目排行榜页面——列出全期和近半年 Star 最高的 50 个仓库，支持按领域/语言筛选，点击卡片展开深度分析。从数据抓取到上线部署，全链路自动化。

为什么叫"纯静态"？

浏览器打开页面后，不再向任何服务器发请求。所有数据（100 个仓库的信息）已经提前写好、塞在 HTML 里了。这和常见的"前端调后端 API"模式完全不同——页面本身就是一个自给自足的完整产品。

新手解读

什么是"静态网站"？ 你平时刷的淘宝、微博，每次打开页面都会向服务器要数据（又叫"动态渲染"）。静态网站恰恰相反：所有内容在部署之前就已经生成好了，服务器只是把文件原样"递"给你。好处是极快（不需要等数据库查询）、极稳（没有后端崩不崩的问题）、极便宜（GitHub Pages 免费）。这个项目就是静态网站的典型代表。

二 数据层：Python + GitHub API

数据从哪里来？

GitHub 提供了一个叫 Search API 的接口，允许你像搜索引擎一样查询仓库。我们用两条查询就够了：

  • stars:>10000 sort:stars → 全时期 Star 最高的 50 个仓库
  • created:2026-01-01..2026-06-18 sort:stars → 近半年创建的新仓库中 Star 最高的 50 个

fetch_data.py 做了什么？

这个脚本（Python）做的事情非常直接：

# 步骤 1：带着 Token 调 GitHub API
resp = requests.get("https://api.github.com/search/repositories",
                     headers={"Authorization": f"token {TOKEN}"},
                     params={"q": "stars:>10000", "sort": "stars", "per_page": 50})

# 步骤 2：从返回的 JSON 中只取我们需要的字段
compressed = []
for repo in resp.json()['items']:
    compressed.append({
        'name': repo['full_name'],      # 仓库名，如 "freeCodeCamp/freeCodeCamp"
        'stars': repo['stargazers_count'],  # Star 数
        'desc': repo['description'],    # 项目描述
        'lang': repo['language'],       # 主要编程语言
        'created': repo['created_at'][:10],  # 创建日期（只取年月日）
        'topics': repo['topics'][:10],  # 标签（最多保留 10 个）
        'stars_month': round(stars / months),  # 计算月均 Star 增速
        # ... 还有 url、forks、license 等
    })

# 步骤 3：如果之前有中文翻译，保留不覆盖
if os.path.exists('data/alltime.json'):
    读旧文件 → 匹配 desc_cn → 写入新文件

# 步骤 4：保存到 data/ 目录
json.dump(compressed, open('data/alltime.json', 'w'), indent=2)

关键字段速查

字段含义用途
name仓库全名（owner/repo）卡片的项目标题
starsStar 总数排序依据 + 进度条宽度 + 统计面板
desc / desc_cn项目描述（英文/中文）卡片副标题 → 展开详情的主要来源

新手解读

JSON 是什么？ JSON（JavaScript Object Notation）是一种纯文本的数据格式，长得像 Python 的字典。它是互联网上最通用的数据交换格式——GitHub API 返回的是 JSON，我们保存的也是 JSON，浏览器里的 JS 直接就能读。它比 Excel 更适合程序处理，比数据库更轻量。

为什么要"数据与展示分离"？ 在这个项目里，data/ 只管"有什么数据"，docs/ 只管"怎么展示"。当你只需要更新数据时，跑 fetch_data.py 就行；只需要改样式时，改 style.css 就行——两者互不干扰。这是软件工程里最基础的"关注点分离"原则。

三 构建层：Python 脚本生成 HTML

build_site.py 的四步流水线

第一步：数据校验（validate_data）

读入 JSON 后，先检查数据是不是"坏的"——空列表？缺少关键字段？Stars 不是数字？任何一个问题都会让脚本直接报错退出，拒绝生成残缺的页面。

def validate_data(data, label):
    if len(data) == 0:
        raise ValueError(f"{label}: empty dataset")
    for repo in data:
        if 'name' not in repo or repo['name'] is None:
            raise ValueError(f"missing required field 'name'")

第二步：领域自动分类（classify_field）

把仓库名 + 描述 + 标签拼成一个字符串，用关键词匹配判断它属于 9 个领域中的哪一个。比如：

# 如果匹配到 "react"、"vue"、"tailwind" 等 → Web前端
# 如果匹配到 "machine learning"、"pytorch"、"llm" 等 → AI/ML
# 如果匹配到 "docker"、"kubernetes"、"linux" 等 → DevOps
# 都不匹配 → 开发工具（兜底分类）

第三步：匹配洞察（INSIGHTS 字典）

14 个重点仓库（如 freeCodeCamp、awesome、build-your-own-x 等）有人工撰写的三层洞察：深入分析 / 解决的问题 / 核心功能。其余 86 个仓库由 auto_insight() 和 auto_features() 自动提取描述。

INSIGHTS = {
    'freeCodeCamp/freeCodeCamp': {
        'insight': '编程教育界的"免费替代一切付费 bootcamp"...',
        'problem': '编程教育资源要么免费但零散，要么系统但昂贵...',
        'features': '11 个完整的认证课程；交互式编程练习环境...'
    },
    # ... 共 14 个
}

第四步：渲染 HTML

使用 Python 的 f-string（字符串模板），把数据、分类、洞察、CSS 全部拼成一个完整的 HTML 文件。其中数据被序列化为 JS 常量嵌入：

const ALLTIME = [
  {"name":"freeCodeCamp/freeCodeCamp","stars":425341,"desc":"...",...},
  ...共50条
];
const RECENT = [ ...共50条... ];

为什么要塞进 HTML 里？

这是关键设计决策。如果页面每次打开都要调 GitHub API：

  • 用户要等网络请求（慢）
  • API 有频率限制（无 Token 时每小时只能 60 次）
  • GitHub API 可能挂掉（页面跟着崩）
  • 移动端网络不稳定时体验差

把数据内嵌后，页面打开即用，筛选/排序/展开全部在浏览器内存里完成，0 次网络请求。

新手解读

"静态网站生成"是什么？ 它的本质是：用脚本（Python/Node/Go 等）在部署之前把数据 + 模板"编译"成纯 HTML。你看到的 top50.html 不是人手写的——是 build_site.py 自动"打印"出来的。这和你熟悉的 React/Vue 有本质区别：React 是浏览器里"现拼"页面（客户端渲染），我们是提前拼好再给浏览器（预渲染）。好处是绝对速度、绝对稳定性、绝对简单。

数据内嵌 vs 调 API： 想象你要给朋友看 50 本书的清单——你可以每次他问你"第 3 本是什么"就翻一次书架（调 API），也可以直接把清单打印出来给他（数据内嵌）。后者当然更快更靠谱。代价是数据"冻"在生成那一刻，需要更新时重新跑脚本。

四 前端层：HTML + CSS + 原生 JS

零框架依赖意味着什么？

这个项目没有用 React、Vue、jQuery、Bootstrap、Tailwind——什么都没有。浏览器原生支持什么，我们就用什么。好处是：不需要 npm install、不需要打包工具、不需要学框架 API、代码量极少（整个页面约 360 行），任何懂 HTML 的人都能直接改。

CSS 设计系统拆解

青蓝彩色高级灰调色板（oklch CSS 变量）

所有颜色都定义在 :root 的 CSS 变量中，使用 oklch() 色彩空间，绝不使用硬编码色值：

:root {
  --cvs: oklch(0.97 0.006 220);  /* 画布底色：极淡冷灰白 */
  --crd: oklch(0.99 0.003 220);  /* 卡片底色 */
  --tx:  oklch(0.25 0.008 220);  /* 正文色：深冷灰 */
  --tx-d:oklch(0.50 0.010 220);  /* 次要文字 */
  --tx-f:oklch(0.68 0.008 220);  /* 辅助文字（最淡） */
  --bd:  oklch(0.88 0.010 220);  /* 边框色 */
}

这 6 个变量构成了完整的视觉层级——从最亮（卡片）到最暗（正文），中间通过边框和辅助色过渡。所有颜色都偏灰，没有高饱和度的"刺眼"色。

明暗双模自动切换

@media (prefers-color-scheme: dark) {
  :root {
    --cvs: oklch(0.18 0.008 220);  /* 暗色画布 */
    --crd: oklch(0.22 0.006 220);  /* 暗色卡片 */
    --tx:  oklch(0.92 0.005 220);  /* 亮色正文 */
    /* ...全部变量翻转 */
  }
}

浏览器会根据系统设置自动选择浅色或深色模式，什么都不用写——CSS 的 prefers-color-scheme 媒体查询会自动生效。

oklch() 领域色

9 个领域各有一种标识色，用 oklch() 色彩空间生成：

/* AI Agent → oklch(0.60 0.05 250) → 蓝紫色调，但饱和度极低 */
/* AI/ML   → oklch(0.60 0.05 180) → 青色系，同样低饱和 */
/* 教育    → oklch(0.60 0.05 95)  → 黄绿色 */

关键：所有领域色的亮度（0.60）和饱和度（0.05）完全一致，只有色相（hue）不同。这让它们在视觉上"同等重要"，不会某个颜色特别抢眼——符合青蓝彩色高级灰"所有颜色都安静"的理念。

原生 JS 交互逻辑

数据绑定

ALLTIME 和 RECENT 两个 JS 变量在 build_site.py 生成时直接写入 HTML，浏览器加载后立即可用：

function switchTab(t) {
  data = t === 'all' ? [...ALLTIME] : [...RECENT];
  doSort();       // 按当前排序方式重排
  buildOpts();    // 重建筛选器选项
  render(data);   // 重新渲染卡片列表

筛选与排序（纯前端计算）

function apply() {
  let items = data;
  if (lang !== 'all') items = items.filter(r => r.lang === lang);
  if (field !== 'all') items = items.filter(r => r.field === field);
  render(items);  // 筛选后重新渲染，完全在浏览器内存中完成
}

function setSort(type) {
  if (type === 'stars')  data.sort((a,b) => b.stars - a.stars);
  if (type === 'month')  data.sort((a,b) => b.stars_month - a.stars_month);
  if (type === 'fresh')  data.sort((a,b) => new Date(b.pushed) - new Date(a.pushed));
  apply();
}

卡片展开/折叠

function toggleCard(card) {
  card.classList.toggle('open');  // 切换 CSS 类，展开区由 CSS 控制显示
}

展开区默认 display:none，加上 .open 类后变为 display:flex。所有动画都由 CSS 的 transition 处理，JS 只做一件事：切换类名。

新手解读

CSS Grid vs Flexbox：怎么记？ Grid 是"二维表格思维"——你心里有一个行×列的网格，然后把元素放到格子里（就像 Excel）。Flexbox 是"一维排列思维"——元素们排成一排（或一列），自动换行、自动分配空间。这个项目卡片内部用 Grid（rank/name/star 三栏位置固定），标签行用 Flexbox（标签数量不定，自然流淌）。

为什么不用 React？ React 解决的核心问题是"数据变了，页面怎么自动更新"。但这个项目的数据在打开页面后就不再变化——它是只读的。筛选和排序只是从已有的 100 条数据里挑子集，不需要"状态管理"、"虚拟 DOM"、"响应式绑定"。就像你不会用一个电饭煲去热一杯凉水——工具要对场景。这里原生 JS 完全够用，而且代码量比加 React 更少。

五 部署层：Git + GitHub Pages + Actions

Git 基础

Git 是一个版本管理工具，记录每次修改的"快照"。对你来说最常用的命令：

git status              # 看改了哪些文件
git add data/ docs/     # 把改动加入"待提交区"
git commit -m "说明"    # 记录一个版本
git push                # 推送到 GitHub 远程仓库

GitHub Pages 原理

GitHub 免费提供的静态网站托管服务。你把文件放在仓库的 docs/ 目录下，GitHub 自动把它们变成可公开访问的网页：

https://QwX777MCC.github.io/oss-top50/           → docs/index.html
https://QwX777MCC.github.io/oss-top50/top50.html  → docs/top50.html
https://QwX777MCC.github.io/oss-top50/tech-stack.html → docs/tech-stack.html
  • 免费：无流量费、无服务器费
  • 自动 HTTPS：自动配置 SSL 证书（地址栏小锁）
  • 全球 CDN：GitHub 的服务器分布全球，访问速度有保障
  • 实时生效：push 代码后约 1 分钟内更新

GitHub Actions CI/CD 全流程

CI/CD（持续集成/持续部署）的意思是把"数据更新 → 构建 → 部署"这一串操作自动化。这个项目的 Action 配置（.github/workflows/update.yml）：

# 触发方式：手动（在 GitHub 网页上点按钮）
on:
  workflow_dispatch:
    inputs:
      reason:  # 可选填"为什么更新"

# 执行步骤（在 GitHub 提供的免费虚拟机上跑）：
steps:
  1. checkout       # 把代码拉下来
  2. setup-python   # 装 Python 3.11
  3. pip install    # 装 requests 库
  4. fetch_data.py  # 调 GitHub API 拿最新数据
  5. build_site.py  # 生成最新 HTML
  6. git commit     # 把变动提交
  7. deploy         # 推送到 gh-pages 分支 → 自动上线

手动触发 vs 定时触发： 当初讨论过两种方案——每天自动跑（cron 定时）和手动点按钮（workflow_dispatch）。最终选了手动，因为：数据变化频率低（一周一次足够）、避免空跑浪费、出错时有人盯着。

SSH 认证

Git 推送代码到 GitHub 需要"自证身份"。我们用的是 SSH 密钥——在你的电脑上生成一对"公钥+私钥"，公钥放 GitHub，私钥留本地。推送时 Git 自动用私钥签名，GitHub 用公钥验证。密钥类型选了 ED25519（比传统的 RSA 更短更快更安全）。

新手解读

CI/CD 到底解决了什么？ 没有它的时候，更新一次榜单需要你手动操作：开终端 → 跑 fetch → 跑 build → git add → git commit → git push。有了 Actions，你只需要在 GitHub 网页上点一下按钮，剩下 7 步全部自动完成。这就是"自动化"——把重复操作交给机器。

为什么选 GitHub Pages 而不是买服务器？ 服务器（云主机/VPS）虽然灵活，但需要你维护操作系统、配置 Web Server（如 Nginx）、管理 HTTPS 证书、监控资源、付月费。对于一个纯静态的排行榜页面，这是用牛刀杀鸡。GitHub Pages 把这些运维成本降到零——你只负责写文件，其他全不用管。

六 设计哲学复盘

形式跟随功能（Form Follows Function）

每个视觉元素的存在都有一个信息传达目的，没有纯装饰性的设计。具体体现：

  • 排名数字不是装饰，是用户的第一入口——所以放在卡片最左，用最大权重
  • 进度条不是图表，是把相对排名可视化——一眼看"它大概排在第几梯队"
  • 领域色不是配色好看，是给大脑一个快速分类的锚点——扫一眼颜色就知道这是 AI 还是 DevOps
  • 展开按钮极弱化（45% 透明度、无边框）——因为它出现 50 次，太抢眼会破坏信息流

青蓝彩色高级灰（oklch 色彩空间）

以冷青蓝（hue 220°）为基调、紫调（hue 285°）为强调的低饱和度色系——"所有颜色都降饱和度、统一亮度"，让页面安静、高级、不累眼。技术上用 oklch() 实现而非传统的 #RRGGBB 或 rgb()：

  • oklch(亮度 饱和度 色相) → 三个参数物理意义独立
  • 传统 rgb/rgba 改一个参数会影响另外两个，调整颜色像"黑盒"
  • oklch 的亮度参数是感知均匀的——亮度 0.6 到 0.7 的视觉变化在所有色相上都一致

F 型阅读流

眼动追踪研究显示，人在网页上的阅读模式呈 F 形——从上到下、从左到右，左侧权重最高。这个页面的卡片布局正是基于 F 型：

┌──────────────────────────────────────────────────────┐
│ [#1]  ← 排名（最左，视线第一落点，视觉权重最高）        │
│                                                      │
│ freeCodeCamp/freeCodeCamp ↗                          │
│ 编程教育界的"免费替代一切付费 bootcamp"...  ← 项目名+描述│
│ [javascript] [react] [nodejs]                  ← 标签  │
│                                   ★ 425,341  ← Star数 │
│                                   ██████████  ← 进度条 │
└──────────────────────────────────────────────────────┘
     左 ════════ 视线流动方向 ════════→ 右（权重递减）

移动端平面构成

768px 断点恢复三栏融合式（rank + info + star 同排），420px 断点 Star 列融入第二行右侧。间距节奏统一基于 4px 网格（gap: 8px, padding: 12px 等），确保视觉上"呼吸均匀"，不出现某处太挤、某处太空的参差感。

新手解读

设计哲学和技术的关系？ 很多人以为设计和代码是两件事——设计师出图，开发者照着做。但在这个项目里，设计决策直接表达在代码结构里。比如"形式跟随功能"变成了一条铁律：每写一行 CSS，都要问自己"这条规则在帮用户理解什么信息？"如果回答不上来，就删掉。这就是"设计工程化"——不是先把视觉做好再写代码，而是用代码来实现设计意图。
